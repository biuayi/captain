# 活动互动平台 — 需求文档

> 本文件在 **check-in-kiosk（前端）** 与 **captain（后端）** 两个仓库中保持同步，作为双方共同的需求基线。
> 创建日期：2026-05-16

---

## 1. 原始需求（用户原文，未加工）

> 我想做一个活动互动的程序。它应该是包含
>
> - 活动现场大屏展示，显示当前 活动ID 的参与人数。这个活动现场大屏，应该是可以具备多款，活动方 在创建一个活动时，可以选择一个 活动大屏。
> - 用户侧：手机端扫码签到 -> 进入对应活动页面，登记信息 -> 小游戏互动。用户侧页面具备多款，活动方 在创建一个活动时，可以选择一个用户侧页面流程（这个流程应该时组合和自定义的）。
> - 后台管理页面（活动方），活动方使用自己的账号密码登录，普通用户无法登录。仅面对授权的活动方。管理页面可以创建活动，设定活动开始时间、结束时间，预计参与人数，活动大屏、用户侧流程。可以通过后台管理页面，看到具体参与用户信息，可以将具体用户参与信息导出。
> - 后台管理页面（超级管理员）：活动方管理，静态资源CDN管理（阿里云OSS）
>
> 服务端应该包含：
>
> - 面对普通用户的活动参与，提供API接口。
> - 面对后台活动方的管理接口
> - 面对超管的管理接口。
>
> 前端仓库为：~/Project/check-in-kiosk/
> 后端仓库为：~/Project/captain/
>
> 数据存储，我倾向使用 pgsql + redis + mq。如果你有更好的推荐，请跟我讲。

## 2. 角色（Roles）

| 角色 | 身份方式 | 说明 |
|---|---|---|
| **普通用户 / 参与者** | 无账号，扫码进入（弱身份待定，见 §7 开放问题） | 仅能参与活动，不能登录任何后台 |
| **活动方 / Organizer** | 授权制账号密码登录 | 仅平台授权的活动方可登录后台，创建并管理自己的活动 |
| **超级管理员 / Super Admin** | 平台账号 | 管理活动方、平台静态资源 CDN |

## 3. 系统组成

### 3.1 前端仓库 check-in-kiosk（React + Vite Monorepo）

三个面向不同角色的 app，外加共享包：

| App | 面向 | 核心职责 |
|---|---|---|
| `big-screen` | 活动现场大屏 | **多款大屏模板**；实时显示当前活动参与人数；活动方创建活动时选其一 |
| `mobile` | 普通用户（H5） | 扫码 → 进入活动页 → 登记信息 → 小游戏互动；**多款、可组合可自定义的用户侧流程** |
| `admin` | 活动方 + 超管 | 后台管理控制台（按角色区分权限与可见功能） |
| `shared` | — | 共享 UI 组件、类型定义、API client、流程渲染引擎 |

### 3.2 后端仓库 captain（Go）

按三类调用方划分接口域：

| 接口域 | 调用方 | 鉴权 |
|---|---|---|
| 参与 API（Participation） | 普通用户（mobile / big-screen） | 匿名 + 扫码 token（防刷，见 §7） |
| 活动方管理 API（Organizer） | 活动方后台 | 活动方账号鉴权 + 多租户数据隔离 |
| 超管管理 API（Admin） | 超管后台 | 超管账号鉴权 |

## 4. 功能需求

### 4.1 普通用户（参与侧）

1. 手机扫码 → 解析活动 ID/token → 进入对应活动的用户侧页面。
2. 按活动方配置的**用户侧流程**逐步进行：典型步骤 `签到 → 登记信息 → 小游戏互动`。
3. 流程是**可组合、可自定义**的：步骤类型、顺序、各步骤配置由活动方在创建活动时编排。
4. 参与行为（签到、登记、游戏结果）写入后端，并实时反映到大屏参与人数。

### 4.2 活动方后台

1. **登录**：账号密码，授权制；普通用户无法登录。
2. **创建 / 管理活动**，至少包含：
   - 活动开始时间、结束时间
   - 预计参与人数
   - 选择一款**活动大屏模板**
   - 选择并配置一套**用户侧流程**（组合 + 自定义）
3. **查看参与用户信息**：活动维度查看每个参与者的登记信息与互动记录。
4. **导出参与信息**：导出该活动的参与用户明细（格式与大数据量导出方式见 §7）。
5. **数据隔离**：活动方只能看到/操作自己的活动与数据。

### 4.3 超管后台

1. **活动方管理**：创建/授权/启用/禁用/编辑活动方账号。
2. **静态资源 CDN 管理**：基于**阿里云 OSS**，管理大屏 / H5 所需静态资源（上传、组织、CDN 分发）。

### 4.4 活动现场大屏

1. **多款模板**，活动方创建活动时选择其一。
2. **实时显示当前活动参与人数**（规模：单场 2000–2万 并发）。
3. 实时方案初定：Redis 计数 + 服务端广播（SSE/WebSocket，见 §6/§7）。

| 维度 | 选型 |
|---|---|

| 单场并发规模 | 中型 **2000–2万** |

## 6. 数据存储建议（回应「如果有更好的推荐」）

- **PostgreSQL ✅**：关系数据（活动方、活动、流程配置、参与记录、登记信息）首选，事务与查询能力足够。
- **Redis ✅**：在本系统承担多职责——大屏实时参与人数计数器、跨实例 SSE 广播（pub/sub 或 Stream）、签到限流与幂等、扫码短 token、活动方会话。是该场景的关键组件，保留。
- **MQ —— 选型建议**：
  - 用途：签到峰值削峰、导出等异步任务、大屏广播扇出、跨服务/跨实例事件。
  - 推荐 **NATS（JetStream）**：Go 生态契合、轻量、运维简单，满足 2000–2万 规模；或 **RabbitMQ**（生态成熟、路由灵活）。
  - **Kafka** 在此规模偏重，除非未来要做大规模行为流分析，否则不必要。
  - 降复杂度选项：初期可用 **Redis Streams** 承担 MQ 职责（少一个组件），规模增长再迁移到 NATS/RabbitMQ。此项留待头脑风暴定。
- **对象存储**：阿里云 OSS（已定）承担静态资源与导出文件存储，配合 CDN 分发。

## 7. 待头脑风暴 / 待确认的开放问题

1. **用户侧流程引擎**：如何"组合 + 自定义"？步骤类型枚举、JSON/DSL 编排 schema、自定义粒度（仅配置参数 vs 自定义页面/分支逻辑）。
2. **大屏实时方案**：SSE vs WebSocket；Redis 计数一致性与重连补偿；多大屏实例。
3. **MQ 选型确认**：NATS / RabbitMQ / Redis Streams。
4. **模板机制**：大屏 / H5 模板是「前端内置组件库 + 配置驱动」还是「远程加载/可上传」；与 OSS 静态资源的关系。
5. **普通用户身份**：完全匿名 vs 手机号/微信 openid 弱身份（影响去重、导出、隐私合规）。
6. **防刷与安全**：扫码 token 生成/时效、签到幂等、限流、防机器人刷量。
7. **多租户隔离**：活动方数据隔离策略（行级 `organizer_id` 过滤 vs schema 隔离）。
8. **数据导出**：格式（CSV/Excel）、大数据量异步导出 + OSS 暂存 + 下载链接。
9. **小游戏互动范围**：内置固定几款 vs 同样模板/插件化；游戏结果如何计入与展示。
10. **部署形态与环境**：环境划分（dev/staging/prod）、单场活动的容量预案、CI/CD。
11. **前后端契约**：API 规范（OpenAPI）、错误码、版本化。

## 9. 头脑风暴结论（codex × Claude，3 轮，2026-05-16）

3 轮头脑风暴已完成，codex 第 3 轮终审「无异议」。最终锁定的工程默认：

| # | 主题 | 最终决策 |
|---|---|---|
| 1 | 架构 | 模块化单体（Go `captain`）；三接口域 participation / organizer / admin 各自独立鉴权域；organizer 与 super-admin **物理分表分域** |
| 2 | 多租户 | 共享 schema + 行级 `organizer_id`，仓储层强制租户过滤 + 防御性校验 |
| 3 | 入场与防刷 | 静态二维码只编码 `event_id + 长期签名 event_token`（标识活动、防篡改，不防重放）；落地后服务端签发 **device-session**（HttpOnly cookie，15min 滑动 idle / 8h 绝对 / `event_end+2h` 硬过期）；幂等键 `submit:{event_id}:{step}:{participant_key}`，`participant_key = identity_hash` 优先否则 `device_uuid`；Redis `SETNX` 快路径 + PG 唯一约束兜底；入口/签发/提交分级限流；被攻击时降级（收紧可疑 IP、入口 JS 挑战、放宽大屏推送到 1/s、Redis 不健康时降级到 PG-only 幂等） |
| 4 | 大屏与计数 | **参与人数 = 完成 `checkin` 步骤的去重 `participant_key` 数**；PG 为事实来源（唯一约束），Redis `INCR` 热计数，参与事件 → Redis pub/sub → SSE **全量快照**（≤2 次/秒节流），每 10s 与 PG 对账纠偏，重连即发最新快照无需回放 |
| 5 | MQ | **NATS JetStream 第一天上**；subjects：`checkin.submitted` / `participant.step_completed` / `export.requested` / `export.completed`；Redis 只管热路径瞬态，JetStream 管持久异步 |
| 6 | 流程引擎 v1 | 四种 step（`checkin` / `form` / `game` / `result`）**线性流，无条件分支**；JSON schema 形状见 ARCHITECTURE.md；`landing` 由入场层处理，不进流程引擎 |
| 7 | 模板 | 内置组件库 + 配置驱动，OSS 只存资源不存可执行包 |
| 8 | 导出 | CSV 异步 → 对象存储 → 短时签名下载链接；`export_job` 表跟踪状态 |
| 9 | 数据模型 | organizer / admin_user / event / flow_config / template / asset / participant / participation / participation_step_record / export_job（字段见 ARCHITECTURE.md） |
| 10 | 工程 | 环境 dev/staging/prod；契约 OpenAPI + `/api/v1` + 显式错误码 |

**M1→M2 最大翻车点（codex 指出）**：流程引擎 JSON schema 与前端渲染解耦不够早。规避：M1 第一天定死 `step_type` 枚举与每种 step 最小 config shape，前端按 `step_type` dispatch 到 renderer，M1 只实现 `checkin` renderer，M2 只填充不改框架。

**第一个垂直切片（codex 建议，作为「初版效果」目标）**：
`扫码 → checkin 完成 → Redis 热计数 +1 → SSE 推大屏 → 后台 event 详情可见参与数 → 触发导出 → 下载 CSV`。
最小表：`event / flow_config(单 checkin step) / participant / participation / participation_step_record / export_job`。

## 10. 三项产品决策（用户授权 Claude 代为确认的默认值，2026-05-16，可后续调整）

> 用户指示「所有的需求细节你帮我确认」，故以下由 Claude 拍板为 v1 默认，记录在案，后续可改。

| # | 决策点 | v1 默认 | 理由 |
|---|---|---|---|
| P1 | 参与者身份策略 | **匿名 `device_uuid` 为主键**；活动方在 `form` 步骤可选采集手机号作为弱身份（采集后 `participant_key` 升级为 `identity_hash`）；**v1 不接入微信 OpenID** | 无需外部集成即可上线；天然支持去重；是否采集手机号交活动方按活动配置；OpenID 需公众号/小程序 appid，推后 |
| P2 | 模板商业模式 | **仅内置组件库**（活动方从内置大屏模板与内置流程 step 组件中选择/配置）；OSS 仅存图片/Logo 等资源，**不允许上传可执行模板包** | 安全攻击面最小、研发可控、上线最快；codex 强烈建议 |
| P3 | 小游戏范围 | **v1 仅内置固定游戏**，首发 `quiz_single_choice`（单选答题）一款，架构保留 `gameType` 扩展位 | 范围收敛保证明早可见效果；插件化为第二期 |

> 章节 §8 原「范围边界」继续生效：支付、抽奖履约、复杂 BI、国际化、App 原生端本期不做。

## 11. 活动方首版具体需求：《寻道大千》周年庆典（2026-05-16 用户补充）

### 11.1 原文

> 活动方身份是 "寻道大千" 游戏……他们希望"实现签到、简单的线上主题游戏互动、还有就是发放礼品（公布奖励页面）、联动公益宣传这样"。他们要举办"寻道大千的周年庆典活动"，面向部分玩家和公司内部员工。其他供应商提供过方案："游戏 IP + 员工互动 + 公益联动"的 H5，包含员工签到闯关、公益步数打卡、多人线上互动小游戏。

### 11.2 IP 主题（联网调研结论）

《寻道大千》：2023 年三七互娱发行的**国风小妖修仙放置 RPG**，黑马爆款。
- 主角：猪 / 鸟 / 鱼 三只妖怪择一；**妖魔鬼怪视角**，非人类主角。
- 美术：**水墨国风**画面，"**丑萌**""有趣"，半写实小妖画风。
- 世界观：前世修仙大能仅剩残魂，今世以小妖之躯重修；腾云驾雾、法宝、灵兽、妖盟、仙途、**大千世界**、寻悟仙道。
- 玩法母题：砍树修炼 → 升级 → 掉装 → 战 BOSS → 升级 循环。
- 来源：见本节末。

**H5 主题基调（供前端落地）**：水墨国风 + 修仙玄幻 + 丑萌小妖（猪/鸟/鱼吉祥物）+ 仙途/大千世界叙事；配色建议墨黑/宣纸米白/朱砂红/竹青/描金点缀。详见 ARCHITECTURE §4 与 `docs/THEME.md`（如有）。

### 11.3 需求拆解与 v1 决策（Claude 代为确认，可调整）

| 诉求 | 映射 | v1 决策 |
|---|---|---|
| 签到 | `checkin` step | ✅ 已支持，去重计数进大屏 |
| 员工签到**闯关** | 多个 `game` step 顺序串联 | ✅ 用线性流的连续 game step 表达"闯关"，不引入分支引擎 |
| 简单线上主题游戏互动 | `game` step（寻道大千主题） | ✅ v1 单人主题游戏（首发：寻道大千主题单选答题 `quiz_single_choice`）；**多人实时小游戏推迟到 v1.x**（实时多人状态同步复杂度高，且用户原话为"简单"，"多人"来自他厂方案描述，非硬指标）→ 待 codex review 确认 |
| 发放礼品（公布奖励页面）| **新增 `reward` step** | ✅ 公布奖励/礼品页：奖励文案、图片、兑换码/领取说明；v1 不做履约发货与库存 |
| 联动公益宣传 | **新增 `charity` step** | ✅ 公益宣传页 + 一键参与/承诺按钮；**公益步数打卡推迟到 v1.x**（需微信运动授权，超出 v1 匿名身份范围）→ 待 codex review |
| 面向玩家 + 公司内部员工 | 同一活动，身份匿名+可选登记 | ✅ `form` step 可选采集姓名/工号区分员工，§10-P1 默认沿用 |

### 11.4 流程引擎 step 类型（据此扩展，覆盖 ARCHITECTURE §4）

v1 线性 step 类型扩为 6 种：`checkin` / `form` / `game` / `charity` / `reward` / `result`。仍无条件分支。

### 11.5 明早「初版效果」种子数据

后端启动种子一个 **「寻道大千 · 周年庆典」demo 活动**，流程：
`签到(checkin) → 寻道大千主题答题(game) → 公益宣传打卡(charity) → 公布奖励(reward)`，
并附一个**最小水墨主题 H5 demo 页**（后端内嵌静态页，仅为可视化验收，后续由 React 前端替换）+ 大屏实时人数页。

### 11.6 推迟项（v1.x，需单独设计/与 codex 头脑风暴）

- 多人实时互动小游戏（状态同步、房间、匹配）
- 公益步数打卡（微信运动/健康数据授权与合规）
